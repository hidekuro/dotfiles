# PR 作成スキル

## このスキルが目指すこと

PR 本文の読者はコードレビュアーです。レビュアーはすでに差分を読めます。だから PR 本文に「○○関数を追加した」「△△ファイルを修正した」と書いても意味がありません。

レビュアーが本当に必要としているのは **コードからは読み取れない文脈** です:

- なぜこの変更が必要だったのか（動機・背景）
- どういう判断でこのアプローチを選んだのか
- どの範囲に影響があるか
- 何を検証済みで、何を検証していないか

このスキルはそこを意識した PR タイトルと本文を生成します。

---

## フェーズ 1: 情報収集

以下を **並列** で取得します。

```bash
# ブランチ・状態の確認
git branch --show-current
git status --short
git rev-parse --abbrev-ref @{upstream} 2>/dev/null || echo "(no upstream)"
gh repo view --json defaultBranchRef -q .defaultBranchRef.name
```

```bash
# 差分の取得（まずサマリーから）
BASE=$(gh repo view --json defaultBranchRef -q .defaultBranchRef.name)
CURRENT=$(git branch --show-current)
git diff --stat ${BASE}...${CURRENT}
git log ${BASE}..${CURRENT} --oneline --decorate
git log ${BASE}..${CURRENT} --pretty=format:"%H%n%s%n%b%n---" --reverse
```

**前提チェック（いずれかに該当する場合は中止）:**

| 状況 | 対応 |
|------|------|
| カレントブランチ = デフォルトブランチ | PR 作成を中止してユーザーに通知 |
| 未コミットの変更がある | コミットを促す（`git status` を提示） |
| 差分が空（コミット 0 件） | PR 作成を中止してユーザーに通知 |

差分が大きい場合（変更ファイル数 > 20 程度）は `git diff --stat` のサマリーで概要を掴み、主要ファイルのみ `git diff` で詳細確認します。

---

## フェーズ 2: リポジトリ規約の確認

**PR テンプレートを探す**（Glob ツールで以下を順にチェック）:

1. `.github/pull_request_template.md`
2. `.github/PULL_REQUEST_TEMPLATE.md`
3. `.github/PULL_REQUEST_TEMPLATE/*.md`（複数ある場合は最初のファイル）
4. `docs/pull_request_template.md`
5. `pull_request_template.md`（リポジトリルート）

テンプレートが見つかった場合は Read ツールで読み込みます。

**コーディング規約を確認**（存在する場合のみ読む）:

- `.github/CONTRIBUTING.md` または `CONTRIBUTING.md`
  - PR タイトルの命名規則（conventional commits 等）
  - 必須チェック項目
  - 言語（日本語 or 英語）の指定

---

## フェーズ 3: 差分の分析（ここが核心）

コミットメッセージと実際の差分を突き合わせながら、以下の問いに答えます。これが PR 本文の品質を左右します。

### 変更の「意図」を読む

**コミットメッセージから読む:**
- コミットが 1 件ならそれが変更の主旨
- 複数のコミットがある場合、共通するテーマを抽出する
- `fix`, `wip`, `temp`, `typo` などの雑なメッセージは表面的なもの。差分の中身で意図を判断する

**差分から読む:**
- **テストコードの変更**: どんな振る舞いが追加・変更されたかを示す。テストの説明文から「何が保証されたか」を読み取る
- **設定ファイルの変更**: デプロイ・環境への影響を示す。本文で明記が必要
- **スキーマ・マイグレーションの変更**: データ構造への影響。特に注意して言及
- **依存関係の変更（package.json 等）**: 外部パッケージの追加・削除。バージョンアップの場合は理由を言及
- **ファイル構成の変化（移動・削除）**: リファクタリングの可能性。既存コードへの影響範囲を言及

**変更の性質を判断:**

| 差分のパターン | 変更の性質 |
|----------------|------------|
| 新規ファイル多数 + テスト追加 | 新機能の追加 |
| 既存コードの書き直し、テストは変化なし | リファクタリング |
| 特定のエラーパスへの修正 + バグ再現テスト追加 | バグ修正 |
| 設定・定数の変更のみ | 設定変更 |
| 複数の無関係な変更が混在 | 雑多な変更（本文でそれぞれ簡潔に言及） |

### 影響範囲を把握する

差分のファイルパスから、変更が及ぶレイヤーを特定します:

- `src/` や `lib/` → ビジネスロジック
- `app/` や `pages/` → UI / エンドポイント
- `tests/` や `spec/` → テストコード（影響範囲外でカウント）
- `infra/`, `deploy/`, `k8s/` → インフラ・デプロイ設定（影響範囲として明記）
- `db/`, `migrations/` → データベース（特に明記が必要）
- `docs/` → ドキュメント

## フェーズ 4: PR タイトルの生成

### タイトルの原則

- **70 文字以内**
- PR 一覧で並んだときに意味が伝わること（「Update files」は NG）
- 変更の「結果として何が実現されるか」を書く

### 命名規則の検出

直近のコミット履歴（`git log --oneline -20`）を確認し、conventional commits 形式（`feat:`, `fix:`, `docs:` 等）が使われているかを確認します。使われていれば PR タイトルにも付与します。

日本語タイトルを使うかは CONTRIBUTING.md の指定、またはリポジトリの言語的文脈から判断します。

### タイトルのパターン

| 変更の性質 | タイトル例 |
|------------|-----------|
| 新機能 | `feat: Add OAuth 2.0 login` / `OAuth 2.0 ログイン機能の追加` |
| バグ修正 | `fix: Resolve memory leak in cache module` / `キャッシュのメモリリーク修正` |
| リファクタリング | `refactor: Extract payment logic into service class` |
| 設定変更 | `chore: Update Node.js version to 20` |
| 複数の変更が混在 | 最も影響が大きい変更を主題にする |

---

## フェーズ 5: PR 本文の生成

### テンプレートがある場合

テンプレートの **全セクションを埋めます**。以下の点に注意:

- セクションの見出し・順序はテンプレートのままにする
- AI が生成した案は **すべて HTML コメント**（`<!-- ... -->`）として記述する。プレースホルダーや既存の `<!-- -->` コメントも AI 提案コメントに置き換える（削除しない）
  - コメントの中身はそのままコメントアウトを解除して使えるように、プレフィックスや括弧は付けない
  - 例: `<!-- このPRはキャッシュのメモリリークを修正します。原因は○○で... -->`
- チェックリスト（`- [ ]`）も含め、AI が書く内容はすべて HTML コメントに収める。人間がコメントを解除して確認する
- テンプレートに書かれていないことは無理に追加しない

### テンプレートがない場合のデフォルト構造

AI が分析した内容はすべて HTML コメントとして挿入する。人間がコメントを確認・編集して実際の本文に昇格させる運用を前提とする。

```markdown
## 概要

<!-- なぜこの変更が必要だったのか。解決する問題・実現したい目的を 2〜4 文で説明する。 -->

## 変更内容

<!--
- 採用したアプローチの選択理由や、コードからは読み取れない判断
- ...
-->

## 影響範囲

<!-- 影響するシステムの範囲。特にDB・インフラ・外部 API など注意が必要な箇所。 -->

## テスト

<!--
- [ ] 既存テストがパスすることを確認した
- [ ] 追加したテストがある場合: 何を検証するテストを追加したか
- [ ] 手動確認が必要な場合: どんな操作で何を確認したか
-->

## 関連 Issue / PR

<!-- あれば記載。なければセクションごと削除 -->
```

### 書き方の注意点

**書くべきこと:**
- 変更の動機（なぜ今これが必要か）
- 採用したアプローチの選択理由（他の選択肢があった場合）
- レビュアーに特に見てほしいポイント
- 既知の制約・TODO・リスク

**書かないこと:**
- コードを見れば明らかなこと（「〇〇メソッドを追加した」「△△を削除した」）
- ファイル名の羅列
- コミットログのコピー
- AI 固有の言い回し（「タスク 1」「フェーズ A」「ステップ 1」等）

**文体:**
- 「である調」（例: 「追加した」「修正した」）または英語で統一
- CONTRIBUTING.md の指定がある場合はそれに従う

---

## フェーズ 6: ユーザーへの提示と確認

本文の大半は HTML コメントになるため、ユーザーへの確認は **タイトルのみ** にします。

```
以下のタイトルで PR を作成します。

**タイトル:** [生成したタイトル]
**ベースブランチ:** [base] ← [current]
**種別:** ドラフト PR（通常 PR にする場合はその旨お知らせください）

よろしければ「OK」または「作成して」と返信してください。
タイトルを変えたい場合はご指示ください。
```

### 修正要求への対応

タイトルの修正要求が来た場合は、修正したタイトルを再提示します。「OK」が出るまでこのループを繰り返します。PR 作成コマンドを実行するのは、明示的な承認が得られた後のみです。

---

## フェーズ 7: PR の作成

ユーザーの承認後、以下の手順で実行します。

### 1. リモートへのプッシュ（upstream がない場合）

```bash
git push -u origin <current-branch>
```

### 2. PR 作成

```bash
gh pr create \
  --title "<タイトル>" \
  --body "$(cat <<'PRBODY'
<本文>
PRBODY
)" \
  --base <base-branch> \
  --assignee @me \
  --draft
```

**オプションの使い分け:**

| オプション | 使用条件 |
|-----------|---------|
| `--assignee @me` | 常に付与（作成者を自動でアサイン） |
| `--draft` | デフォルト。ユーザーが「ドラフトではない」と明示した場合は省略 |
| `--reviewer` | ユーザーが指定した場合のみ |
| `--label` | ユーザーが指定した場合のみ |

### 3. 完了通知

作成完了後、PR URL を提示します:

```
PR を作成しました:
https://github.com/...
```

---

## エラーハンドリング

| エラー | 対処 |
|--------|------|
| リモートブランチが存在しない | `git push -u origin <branch>` で先にプッシュ |
| `gh` が未認証 | ユーザーに `gh auth login` の実行を促す |
| 同名の PR が既に存在する | 既存 PR の URL を提示し、更新するか確認 |
| ベースブランチが存在しない | ユーザーに正しいベースブランチを確認 |

---

## 制約事項

- **PR 作成後の自動操作は行わない**: マージ・レビュアー割り当て等は明示的な指示があった場合のみ
- **TodoWrite や Agent ツールは使用しない**: Bash, Read, Glob, Grep のみを使用する
- **並列実行を活用する**: 独立した情報取得は並列で実行して効率化する
- **リポジトリ規約を優先する**: PR テンプレートや CONTRIBUTING.md の指定は必ず従う
