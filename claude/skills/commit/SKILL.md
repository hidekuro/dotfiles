# Commit

## 手順

1. `git status` と `git diff` で変更の全体像を把握する
2. 変更を論理的なまとまりでコミット単位に分割し、各コミットのメッセージ案をユーザーに提示して確認を求める
3. ユーザーの確認後に `git add` + `git commit` を実行する

## コミットメッセージの書き方

コミットメッセージは「次に読む人が変更の動機を理解できるか」を基準に書く。

**絶対禁止: 変更内容（Whatの列挙）をそのまま書くこと**

差分を見ればわかることは書かない。「何を変更したか」ではなく「なぜその変更をしたか・何のためにしたか」を書く。

| 禁止（Whatの列挙） | よい（Whyにフォーカス） |
|---|---|
| `○○メソッドにエラーハンドリングを追加` | `○○処理が失敗したときにユーザーにエラーを伝えられなかった問題を修正` |
| `キャッシュ機構を導入し Redis を使うよう変更` | `ユーザー増加による応答遅延を解消するためキャッシュを導入` |
| `foo.ts と bar.ts を修正` | `（差分が小さければ1行で動機を書く）` |

**Whyが差分から読み取れないとき:**

- 会話の文脈（ユーザーの依頼内容・背景）から推論して書く
- 文脈からも不明なときはユーザーに動機を確認してから書く

**形式:**

```
<1行目: 変更の目的・動機を表す概要>

<3行目以降: コードから読み取れない背景・制約・意図 (必要な場合のみ)>
```

- 1行目は「である調」（レポート調）で書く（「〜する」「〜した」など）
- 3行目以降が必要なのは、コード差分が大きいか、動機の説明が複雑なとき。それ以外は1行目だけにする
- 本文を書くときも簡潔にし、動機が伝わる1〜2文に収める。冗長な経緯説明や、コードを見ればわかる変更内容の列挙はしない
- 文の途中で改行しない。1文=1行で書き、行幅に合わせた折り返しを入れない（1行あたりの文字数は無制限）
- 並列構造があるときは `- ` で始まるリスト形式にする
- AI エージェント特有の用語（「タスク1」「フェーズ1」など）は使わない

**AI エージェントとして実行している場合:**

コミットメッセージの末尾（空行の後）に `Co-Authored-By` トレーラーを付与する。モデル名はシステムコンテキストから読み取った実際のモデル名（例: `Claude Opus 4.6`、`Claude Sonnet 4.6` など）を使うこと。

```
Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

## 例

**例 1 — 概要のみ（差分が小さい・動機が自明なとき）:**

```
○○処理が失敗したときにユーザーにエラーが伝わらなかった問題を修正
```

**例 2 — 背景説明あり:**

```
ユーザー増加による応答遅延を解消するためキャッシュを導入

ユーザー数増加により応答時間が3秒を超えるケースが発生。
頻繁にアクセスされるデータを Redis でキャッシュすることで平均応答時間を 500ms 以下に短縮する。
```

**例 3 — AI エージェントが実行した場合:**

```
ユーザー増加による応答遅延を解消するためキャッシュを導入

ユーザー数増加により応答時間が3秒を超えるケースが発生。
頻繁にアクセスされるデータを Redis でキャッシュすることで平均応答時間を 500ms 以下に短縮する。

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```
