# Commit

## 手順

1. `git status` と `git diff` で変更の全体像を把握する
2. 変更を論理的なまとまりでコミット単位に分割し、各コミットのメッセージ案をユーザーに提示して確認を求める
3. ユーザーの確認後に `git add` + `git commit` を実行する

## コミットメッセージの書き方

コミットメッセージは「次に読む人が変更の動機を理解できるか」を基準に書く。コードを読めばわかる変更内容の説明は不要で、なぜその変更をしたか・何のためにしたかを書く。

**形式:**

```
<1行目: 変更の目的・動機を表す概要>

<3行目以降: コードから読み取れない背景・制約・意図 (必要な場合のみ)>
```

- 1行目は「である調」（レポート調）で書く（「〜する」「〜した」など）
- 3行目以降が必要なのは、コード差分が大きいか、動機の説明が複雑なとき
- 3行目以降は `- ` で始まるリスト形式を使ってもよい
- 行幅を意識した改行は不要（1行あたりの文字数は無制限）
- AI エージェント特有の用語（「タスク1」「フェーズ1」など）は使わない

**AI エージェントとして実行している場合:**

コミットメッセージの末尾（空行の後）に `Co-Authored-By` トレーラーを付与する。

```
Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

## 例

**例 1 — 概要のみ（差分が小さい・動機が自明なとき）:**

```
不足していたエラーハンドリングを追加
```

**例 2 — 背景説明あり:**

```
パフォーマンス改善のためキャッシュ機構を導入

ユーザー数増加により応答時間が3秒を超えるケースが発生。
頻繁にアクセスされるデータを Redis でキャッシュすることで平均応答時間を 500ms 以下に短縮する。
```

**例 3 — AI エージェントが実行した場合:**

```
パフォーマンス改善のためキャッシュ機構を導入

ユーザー数増加により応答時間が3秒を超えるケースが発生。
頻繁にアクセスされるデータを Redis でキャッシュすることで平均応答時間を 500ms 以下に短縮する。

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```
