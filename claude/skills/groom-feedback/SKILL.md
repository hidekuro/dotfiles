# Groom Feedback

## 目的

フィードバックはメモリに溜まる一方、本来の格納先 (スキル・CLAUDE.md) に反映されないと二重管理になり、同じ指摘が繰り返される。このスキルは全プロジェクトのフィードバックメモリを横断で棚卸しし、次の 2 つを行ってズレを解消する。

- 繰り返される手順や作法を**新しいスキルへ昇華**する
- 既存スキル・CLAUDE.md の**改善**につなげる

起動したプロジェクトに依存しない。すべてのプロジェクトの memory を対象にする。

## 前提: スキルのオリジン

インストール済みスキル (`~/.claude/skills/`) は dotfiles (`~/.dotfiles/claude/skills/`) からコピーされたもの。スキルを新規作成・編集するときは必ずオリジンを編集し、dotfiles リポジトリでコミットする。インストール版にも同じ内容をコピーし、即座に使える状態にする。

## 手順

1. **全プロジェクトのフィードバックを収集する**
   - `~/.claude/projects/*/memory/` 配下の `type: feedback` メモリを列挙する (各ディレクトリの `feedback_*.md` と `MEMORY.md` の索引)。
   - プロジェクトをまたいで同じ主題のフィードバックはまとめて 1 件として扱う (例: コミットメッセージに関する指摘が複数プロジェクトに散っている)。

2. **格納先を判定する**
   - 特定スキルの挙動に関する指摘 → そのスキルの改善
   - 単発では小さいが**繰り返し現れる手順・作法**で、既存スキルのどれにも当てはまらないもの → 新しいスキルへ昇華する候補
   - プロジェクト共通の一般的な作法 → CLAUDE.md (グローバルまたはプロジェクト)
   - セッション固有・一時的なもの → メモリのまま (反映しない)

3. **未反映を検出する**: 判定した格納先に既に反映されているか確認する。反映済みなら触らない。

4. **反映案を作る**: 未反映のものについて、次のいずれかの案を作りユーザーに提示して確認を求める。
   - 既存スキル・CLAUDE.md への追記・修正
   - 新しいスキルの新規作成 (`skill-creator` を使う。名前はユーザーに候補を出して選んでもらう)

5. **スキル変更時は検証する**: スキルを新規作成・変更した場合は `skill-creator` で旧版スナップショット (新規なら未使用時) と新版に同じ入力を流し、退行や過剰反応がないか確認する。変更が軽微なら省略してよい。

6. **コミットする**: 承認後、オリジンを編集していれば dotfiles リポジトリで commit スキルに従ってコミットし、インストール版へコピーする。

7. **メモリを後始末する**: 恒久先へ完全に吸収され単独のリマインダーとして不要になったメモリは削除する。参照価値が残るものはそのまま残す。各プロジェクトの `MEMORY.md` の索引も更新する。

## 注意

- 全フィードバックがスキル行きではない。手順 2 の判定を飛ばさない。
- 反映は最小限・具体的に。スキル本体を肥大化させない。
- ユーザーの確認前に編集・コミットしない。
- 新スキルの昇華は「繰り返し現れる」ことが根拠。1 件だけの指摘で安易にスキルを作らない。
