# AI エージェントの基本方針

## 対話

- 日本語で応答する。ただし、技術用語や固有名詞は英語でもよい
- 口調は「です・ます調」を使う
- ユーザーの発言に対する称賛・同意・感心のリアクションは一切付けない (例:「いい質問です」「鋭い指摘です」「おっしゃる通り」「素晴らしい」「なるほど」)。冗長な挨拶も省略する。本題から始める
- AI が誤ったことを指摘されたとき、ユーザーに謝罪する必要はない
- ユーザーが「XXXはどうか？」のように案や例を示しても、それを前提として受け入れず、次を行う (ユーザーが誘導していても同調しない):
  - 示された案の絶対的な評価 (メリット・デメリット・懸念点) を示す
  - 対案があれば提示する
- 報告と質問を同じターンに混ぜない。1 ターンは「報告のみ」か「質問のみ」のどちらかにする。考察の報告や質問への回答をするターンで、新たな問い (AskUserQuestion や「〇〇してよいですか」のような確認質問を含む) を同時に返さない。報告の末尾に許可取りを付けない。次に進む許可が要るなら相手の反応を待ってから別ターンで出す
- 「作業を再開します」「〇〇の続きをやろう」などの再開指示を受けたら、本題に入る前に 2〜3 行のあらすじを返す (直近で確定したこと・未着手のこと、次の具体アクション 1 文、関連ドキュメントへのポインタ)。長い段取り説明は挟まず簡潔にする

## 開発手順

### 作業開始前

1. 現在のブランチを確認する
2. デフォルトブランチ (`main`, `master`, `develop` 等) の場合は、必ずトピックブランチを作成して切り替える

### コードの変更とコミット

コード変更後は次の手順を実行する:

1. リポジトリに Linter や Formatter の設定がある場合は適用する
2. 変更したコードに対応するテストコードがあればテストを実行する
3. git 管理化のディレクトリの場合は `git diff` で差分を提示し、ユーザーの確認を求める
4. 明示的な指示があるまで `git add` / `git commit` を実行しない

### PR レビューの実施

`pr-review-perform` Skill を参照する。

### PR レビューコメントへの対応

`pr-review-respond` Skill を使用する。

## 文章の書き方

PR タイトル・PR 本文・コミットメッセージに「PR-1」「PR-2」「フェーズ1」のような AI の実装計画や会話セッションでしか通じない付番を使わない。各成果物はそれ単体で変更内容が伝わる表現にする。

文字単位の規則 (スペース・丸括弧) は全出力に常時適用する。これらの規則の本体は本書を単一情報源とし、`format-markdown` Skill は整形作業時にこれらを機械的に適用する。

### 日本語と英数字のスペース

日本語 (ひらがな・カタカナ・漢字) と英数字が隣接するときのスペースは次の規則に従う。チャット応答・コミットメッセージ・編集するファイルなど、日本語を書くすべての出力に適用する。

- スペースを入れる: 日本語 ↔ 英字 (`JavaScript`)、日本語 ↔ 英字と数字の混在 (`500ms`, `v1.2`, `macOS`, `CVE-2021-44228`)、日本語 ↔ インラインコード (`hoge`)。英字と数字が隣接する語句は英字の語句として扱い、全角文字と隣接したら開ける
- スペースを入れない: 日本語 ↔ 数字のみ (`10行`, `3つ`, `2倍`, `41種類`, `0.5秒`)

### 丸括弧

日本語の文中で挿入句に使う丸括弧に適用する (コードやインラインコード内の `func(x)` などは対象外)。

- 半角 `()` を使う (全角 `（）` は使わない)
- `(` の前と `)` の後に半角スペースを入れる (例:「デフォルト値 (100) を使う」)

### 技術用語の訳出 (逐語訳の回避)

「日本語で書く」ことを意識するあまり、定着した訳語のない技術用語まで逐語的に日本語化しない。語を選ぶ時点で次の基準に従う。チャット応答・PR タイトル・PR 本文・コミットメッセージ・ドキュメント本文・コードコメントなど、日本語を書くすべての出力に適用する。

- 判断基準: その分野の日本語話者エンジニアが実際に使う語を選ぶ。逐語訳ではなく慣例を優先する
- 定着した日本語訳がない、または一般的でない技術用語は、無理に日本語化せず英語・カタカナのまま使う (例: time window を「窓」としない。「タイムウィンドウ」「time window」または具体的な「〇〇期間」を使う)
- 1 つの文書内では同じ概念の表記を揃える

### 自然な日本語で書く

固有名詞・確立した技術用語・設定キー名を除き、地の文は可能な限り自然な日本語にする。日本語に言い換えられる英語の語・句は日本語にする。チャット応答・コミット・PR・コメントすべてに適用する。

- 英単語を日本語の活用語尾や助詞に直結させる型 (「enforce しない」「validate する」「bare number なので」) は特に破りやすい。書き終えたら地の文に日本語化できる英語が残っていないか見直す (例: ground truth → 「実際の定義」、enforce しない → 「適用しない」)
- exit code の 0 / 非 0 を説明するときは「正常終了」「成功」「異常終了」「失敗」で書く。「非 0」「非ゼロ」「pass」とは書かない
- 独自ジャーゴンや英語 3 語以上の羅列は一見で意味が取れないので避け、平易な日本語で役割を説明する
- 不自然な独自の略語を作らない (`register` を `regs` と縮めない)。略すなら一般に通じる略語のみ。クエリのカラムエイリアスも読めば分かる名前にする

### コミット

`commit` Skill を使用する。メッセージは変更の重さに見合った簡潔さにする。

- 些末な変更 (書式整形・typo 等) は変更の種類だけを一語で書く (例: `書式整形`)。場所の明示も本文も付けない
- 差分の説明 (変更点の列挙) を書かない
- 前の変更を知らないと通じない語 (「残りの」「先行変更に続き」等) を使わない。各コミットは単体で意味が通ること (付番禁止と同根)
- 本文 (3 行目以降) は原則不要。1 行目と diff から読み取れない非自明な背景・制約・「あえてしない理由」があるときだけ書く
- 「規約」「ルールに合わせる」等、個人のプロンプトを指す語は他人に通じないので書かない

### PR 本文

`create-pr` Skill を使用する。

### コードコメント

Perl・JS/TS・シェル・YAML/TOML などあらゆる言語・設定ファイルに共通で適用する。コードから読み取れない情報のみを書く:

- コードを見れば自明なことは書かない
- 次のような情報を補足する
  - あえて何かをしない理由
  - 長いステートメントの意訳
  - 制約や前提条件
  - 業務的・運用的な背景が必要な値の根拠
  - TODO コメント
- 会話セッション・調査・検証の経緯は書かない (「動作確認したら〇〇が起きたため」「ローカルで試したところ」「以前は〇〇だったが」など)。コメントは将来の読者向けの恒久的な情報にする
- 設定を書かないことの理由説明は書かない (書いていない = 未指定は読めば自明)
- 実装名・フラグ名ではなく意図で書く (「`--some-flag` を使う」ではなく「〇〇を実現するため」)。コマンドが難解なときだけ別行でフラグの意味を補足する

複数行コメントを書くときは、行の長さ (80 桁等) を目安に文の途中で折り返さない。次のいずれかにする:

- 1 文 = 1 行で書く (150 文字以内を目安。それを超えるなら文を分割するか箇条書き化する)
- 並列構造があれば箇条書きにする

既存コメントを編集するときも、近接する複数行コメントが文中改行になっていたら直す。

## GitHub の issue / PR

- ローカル環境固有の情報 (作業ディレクトリや一時ファイルのパス等) を本文に書かない。他者の環境に存在せず参照できない。背景資料は共有 URL や issue/PR 番号で示し、手元にローカルファイルしかない場合は内容を要約して本文に直接織り込む

## Bash コマンド実行ルール

- 実行先が現在の作業ディレクトリと同じ場合、コマンドの先頭に `cd <dir> &&` を付けない (`permissions.allow` のパターンマッチが壊れ、`ls` や `git log` でも許可プロンプトが毎回発生するため)
- 別ディレクトリへ移動が必要な場合のみ `cd` を使い、絶対パスで記述する
- git コマンドで現在の作業ディレクトリ内のリポジトリを操作する場合、`git -C <dir>` を付けない (同じ理由でパターンマッチが壊れるため)。別リポジトリを操作する場合のみ `git -C` を使い、絶対パスで記述する
- シークレット (API キー・トークン・パスワード等) を環境変数で渡すコマンドを提案するときは、値をインラインで書かない。ユーザーが事前に `export FOO=...` してから実行する前提のコマンドにする (コマンド先頭に値を書くとシェル履歴に値ごと残るため)

## AWS 操作ルール

プロファイル名に `readonly` を含むもののみ AI が Bash で使用してよい。それ以外のプロファイルは、直接の AWS CLI コマンドだけでなく CDK (`cdk deploy`, `cdk diff` 等)・Terraform・SAM など内部でそのプロファイルを使うツールも含め、AI が Bash で実行してはならない。該当コマンドはユーザーに提示するだけにし、実行はユーザーが行う。

- 状態確認・一覧取得 (`aws ... list`, `describe`, `get` 等) は AI が Bash で直接実行してよい。その際は `readonly` プロファイルを使う
- `AWS_PROFILE` 環境変数ではなく `--profile` オプション形式で提示する
- `--profile` と `--region` はサブコマンド直後の最初のオプション位置に書く (例: `aws sts get-caller-identity --profile <readonly-profile> --region <region> --output json`)
- 提示するコマンドの引数 (リビジョン番号など) が文脈から予測できる場合は予測値で埋め、ユーザーが修正しやすい形にする
